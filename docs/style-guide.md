# Writing Style Guide

The audience is a future McGill Rocket Team member who may not be an expert in telemetry, radios, or ground station systems.

## Tone

- Clear, direct, and technical.
- Helpful to beginners without being vague.
- Concise enough that an AI can maintain it without creating repeated filler.

## Content Rules

- Define acronyms on first use.
- Prefer facts over storytelling.
- Use bullets and tables for scanability.
- Keep paragraphs short.
- Record only what is known, what was said, or what is clearly labeled as uncertain.
- Add open questions when the page cannot be completed from the available information.

## Avoid

- Repeating the same explanation across multiple pages.
- Long transcript-like dumps.
- Unsourced guesses.
- Vague claims such as "works well" without context.
- Broad summaries that hide important configuration values or procedures.

## Useful Labels

Use these labels when needed:

- Known: confirmed project knowledge.
- Unknown: information not yet provided.
- Unconfirmed: plausible but not verified.
- Needs decision: a choice still has to be made.
- Historical: true for a past design, test, or setup but not necessarily current.

## Example Wording

Prefer:

> The ground station logs received telemetry packets to local storage during launch operations. The exact file format is unknown.

Avoid:

> The ground station probably saves a bunch of telemetry data somewhere and it should be pretty easy to check later.
